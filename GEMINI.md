# Project GEMINI: AWS Microservice Lab (DynamoDB + ElastiCache + ECS Fargate)

This project is a production-grade lab tutorial for building a scalable microservice on AWS using a **cache-aside** architecture. It focuses on high performance (p99 < 20ms), security (private subnets, no NAT), and observability.

## Project Overview

- **Purpose**: Demonstrate a microservice pattern using DynamoDB as the primary store and ElastiCache Redis as the caching layer, running on serverless ECS Fargate.
- **Main Technologies**:
  - **AWS Services**: DynamoDB, ElastiCache (Redis), ECS Fargate, ALB, VPC Endpoints (PrivateLink), Secrets Manager, CloudWatch, ECR.
  - **Runtime**: Python 3.12 (FastAPI).
  - **IaC**: CloudFormation for networking, security groups, and IAM roles.
  - **Networking Strategy**: Uses VPC Endpoints (Gateway & Interface) to eliminate the need for NAT Gateways, reducing cost and increasing security.

## Architecture Highlights

- **Cache-Aside Pattern**: API checks Redis first; on miss, it queries DynamoDB and populates the cache.
- **Private-Only Compute**: ECS tasks run in private subnets with NO internet egress. All AWS API traffic stays within the AWS backbone via PrivateLink.
- **Multi-AZ HA**: Redis is configured with a replication group across 2 AZs. ALB and ECS tasks are also distributed across 2 AZs.
- **Security**: 
  - Least-privilege IAM roles (separated Task vs. Task Execution).
  - SG-to-SG rules (no CIDR-based ingress where possible).
  - TLS/AUTH for Redis.

## Deployment Modules

The lab is structured into sequential modules:
1. **Module 1**: VPC & Networking (CloudFormation) - Sets up the VPC, subnets, and all necessary Gateway/Interface endpoints.
2. **Module 2**: DynamoDB Table - Creates the `products` table with On-Demand billing and GSI.
3. **Module 3**: ElastiCache Redis - Deploys a Multi-AZ Redis replication group with TLS and AUTH.
4. **Module 4**: Application Code - Implementation of the FastAPI app with `singleflight` protection against cache stampede.
5. **Module 5**: Build & Push - ECR repository setup and Docker image push.
6. **Module 6**: ECS Fargate - IAM Roles, ALB, Task Definition, and Service deployment.
7. **Module 7**: Test & Verify - Validation of cache hits/misses and load testing.

## Critical Commands

### Deployment
- **VPC Setup**: `aws cloudformation deploy --template-file 01-network.yaml --stack-name ddb-cache-lab-network ...`
- **Redis Auth**: `aws secretsmanager create-secret --name ddb-cache-lab/redis/auth-token ...`
- **Docker Push**: `docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/ddb-cache-lab/product-api:v1.0.0`

### Cleanup (Mandatory)
Follow the specific order in **Module 9** of the lab document to avoid orphan resources and unexpected costs. The order is:
1. Scale ECS Service to 0 -> Delete Service.
2. Delete ECS Cluster.
3. Delete ALB & Target Group.
4. Delete ElastiCache Replication Group & Subnet Group.
5. Disable Deletion Protection -> Delete DynamoDB Table.
6. Delete Secrets Manager Secret.
7. Delete ECR Repository.
8. Delete CloudFormation Stacks (IAM, SG, Network).

## Development Conventions

- **Security First**: Never hardcode secrets. Use Secrets Manager injection in Fargate.
- **Efficiency**: Use Graviton (ARM64) for Fargate tasks to save cost.
- **Resilience**: Implement graceful degradation; the app should function (though slower) if Redis is unavailable.
- **Infrastructure as Code**: Prefer CloudFormation or Terraform for resource management over manual Console actions.
